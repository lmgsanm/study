## GOLANG项目的单元测试

### 前提：编写 Go 单元测试

1. **测试文件**：与源码同目录，命名为 `*_test.go`

2. 测试函数

   ：命名 

   ```
   TestXxx(t *testing.T)
   ```

   go

   

   运行

   

   

   

   

   ```
   // math.go
   func Add(a, b int) int { return a + b }
   
   // math_test.go
   package main
   
   import "testing"
   
   func TestAdd(t *testing.T) {
       if Add(1, 2) != 3 {
           t.Error("1+2 should be 3")
       }
   }
   ```

   

3. 运行命令

   （本地）：

   bash

   

   运行

   

   

   

   

   ```
   go test -v ./...          # 跑所有测试
   go test -race ./...       # 检测竞态条件
   go test -cover ./...      # 覆盖率
   ```

   

------

### 二、最简 GitLab CI 配置（推荐）

在项目根目录创建 **`.gitlab-ci.yml`**：

yaml











```
# 1. 使用官方 Go 镜像
image: golang:1.22  # 用你项目的 Go 版本

stages:
  - test

# 2. 单元测试任务
unit-test:
  stage: test
  before_script:
    # 可选：加速依赖下载
    - go env -w GOPROXY=https://goproxy.io,direct
    # 下载项目依赖
    - go mod download
  script:
    # 代码格式化检查（可选）
    - go fmt ./...
    # 静态检查（可选）
    - go vet ./...
    # 核心：运行所有测试（带竞态检测、失败即停）
    - go test -v -failfast -race -coverprofile=coverage.out ./...
    # 查看覆盖率摘要（可选）
    - go tool cover -func=coverage.out
  # 3. 缓存依赖，加速下次构建
  cache:
    paths:
      - .go/pkg/mod/
  # 4. 始终上传产物（即使测试失败）
  artifacts:
    when: always
    paths:
      - coverage.out
    reports:
      # 支持 GitLab 内置测试可视化（需 gotestsum）
      junit: report.xml
```

------

### 三、进阶：生成 GitLab 可视化测试报告

原生 `go test` 不支持 JUnit XML，用 **`gotestsum`** 生成：

yaml











```
unit-test:
  stage: test
  image: golang:1.22
  script:
    - go env -w GOPROXY=https://goproxy.io,direct
    - go mod download
    # 安装 gotestsum
    - go install gotest.tools/gotestsum@latest
    # 跑测试并生成 JUnit 报告
    - gotestsum --junitfile report.xml --format testname -- -race -cover ./...
  artifacts:
    when: always
    reports:
      junit: report.xml  # GitLab 自动解析
```

效果：

- GitLab → 你的项目 → **构建 → 测试报告**
- 可看通过率、失败用例、耗时

### 四、完整企业级配置（含缓存、覆盖率、多阶段）

yaml











```
image: golang:1.22

# 定义缓存模板（复用）
.go-cache:
  cache:
    paths:
      - .go/pkg/mod/
    key:
      files:
        - go.mod
        - go.sum

stages:
  - lint
  - test
  - build

# 1. 代码质量检查
lint:
  stage: lint
  image: golangci/golangci-lint:latest
  extends: .go-cache
  script:
    - golangci-lint run ./... --timeout 5m

# 2. 单元测试（带报告+覆盖率）
test:
  stage: test
  extends: .go-cache
  script:
    - go env -w GOPROXY=https://goproxy.io,direct
    - go mod download
    - go install gotest.tools/gotestsum@latest
    - gotestsum --junitfile report.xml -- -v -race -coverprofile=coverage.out ./...
    - go tool cover -html=coverage.out -o coverage.html
  artifacts:
    when: always
    paths:
      - coverage.html
    reports:
      junit: report.xml

# 3. 构建（可选）
build:
  stage: build
  extends: .go-cache
  script:
    - go build -o app ./cmd/...
  artifacts:
    paths:
      - app
```

------

### 五、关键参数说明

- **`go test -v`**：详细日志
- **`-failfast`**：第一个失败就停（CI 推荐）
- **`-race`**：开启数据竞态检测（必加）
- **`-coverprofile`**：生成覆盖率文件
- **`gotestsum`**：生成 GitLab 支持的 JUnit 报告
- **`cache`**：缓存 `go mod` 依赖，大幅提速
- **`artifacts.reports.junit`**：开启 GitLab 测试可视化

------

### 六、常见问题

1. **测试不失败（panic 也过）**

   

   - 必须加 **`-race`** 或 **`-failfast`**，强制非零退出码
   - 用 `gotestsum` 更可靠

   

2. **依赖下载慢**

   

   - 加 `GOPROXY`：`go env -w GOPROXY=https://goproxy.io,direct`

   

3. **私有依赖拉不到**

   

   - 在 CI/CD 设置中配置 **Git 凭据 / SSH 密钥**

   

4. **报告不显示**

   

   - 确认 `artifacts.reports.junit` 指向正确 XML
   - 检查 GitLab 版本（需 12.4+）



## sonar扫描

在 GitLab CI 中为 Go 项目集成 SonarQube/SonarCloud 代码扫描，核心流程是：**运行单元测试 → 生成测试报告与覆盖率 → 用 SonarScanner CLI 上传分析 → 结果同步回 GitLab MR**。下面是完整可直接复制的配置方案。

### 一、准备工作（管理员 / 配置一次）

#### 1. 在 SonarQube 创建项目

1. 登录 SonarQube → **Create project → GitLab**（或手动）

2. 获取：

   - `Project key` (如 `my-go-app`)
   - `SONAR_TOKEN` (生成令牌，**保密**)
   - `SONAR_HOST_URL` (如 `https://sonar.example.com`)

   

#### 2. GitLab 项目设置 CI 变量

项目 → **Settings → CI/CD → Variables**，添加：

- `SONAR_TOKEN`：值为上面的令牌（**勾选 Protect & Mask**）
- `SONAR_HOST_URL`：SonarQube 地址

------

### 二、项目根目录添加配置文件

#### 1. sonar-project.properties（必选）

properties











```
# 项目标识
sonar.projectKey=my-go-app          # 替换为你的 Project key
sonar.projectName=my-go-app
sonar.projectVersion=1.0

# 源码路径（Go 项目必须设为 .）
sonar.sources=.
sonar.exclusions=**/vendor/**,**/*_test.go  # 排除测试和 vendor

# 测试文件配置
sonar.tests=.
sonar.test.inclusions=**/*_test.go

# Go 覆盖率报告（由 go test 生成）
sonar.go.coverage.reportPaths=coverage.out

# 测试执行报告（可选，需 gotestsum）
sonar.go.tests.reportPaths=report.xml

# 编码与语言
sonar.sourceEncoding=UTF-8
sonar.language=go
```

#### 2. .gitlab-ci.yml（核心流水线）

yaml











```
stages:
  - test
  - sonar

# 缓存 Go 依赖与 Sonar 缓存（加速）
cache:
  paths:
    - .go/pkg/mod/
    - .sonar/cache/

variables:
  # 禁用浅克隆（Sonar 需要完整 Git 历史）
  GIT_DEPTH: "0"
  # Go 模块代理
  GOPROXY: "https://goproxy.io,direct"

# 1. 测试阶段：生成测试报告 + 覆盖率
go-test:
  stage: test
  image: golang:1.22  # 匹配你的 Go 版本
  script:
    - go mod download
    # 安装 gotestsum（生成 JUnit 报告）
    - go install gotest.tools/gotestsum@latest
    # 运行测试：带竞态检测 + 覆盖率 + JUnit 报告
    - gotestsum --junitfile report.xml --format testname -- -v -race -coverprofile=coverage.out ./...
    # 查看覆盖率摘要
    - go tool cover -func=coverage.out
  artifacts:
    # 测试失败也保留报告给 Sonar
    when: always
    paths:
      - coverage.out
      - report.xml
    reports:
      junit: report.xml  # GitLab 内置测试报告

# 2. Sonar 扫描阶段
sonar-check:
  stage: sonar
  image: sonarsource/sonar-scanner-cli:latest
  needs: [go-test]  # 依赖测试完成
  script:
    - sonar-scanner
        -Dsonar.projectKey=my-go-app  # 与 properties 一致
        -Dsonar.token=$SONAR_TOKEN
        -Dsonar.host.url=$SONAR_HOST_URL
        -Dsonar.qualitygate.wait=true  # 等待质量门禁结果
  allow_failure: false  # 质量门禁失败则 pipeline 失败
  # 仅在主分支和合并请求触发
  rules:
    - if: $CI_COMMIT_BRANCH == "main" || $CI_PIPELINE_SOURCE == "merge_request_event"
```

------

### 三、关键配置说明

#### 1. Go 报告生成

- **测试报告**：`gotestsum` → `report.xml`（JUnit 格式）

- **覆盖率**：`go test -coverprofile=coverage.out` → Sonar 原生支持SonarQube

- 必加参数

  ：

  - `-race`：检测数据竞争（Go 必开）
  - `-coverprofile`：覆盖率文件
  - `GIT_DEPTH:0`：Sonar 需要完整 Git 历史SonarQube

  

#### 2. Sonar 关键参数

- `sonar.sources=.`：Go 项目必须设为根目录（含 `go.mod`）SonarQube
- `sonar.exclusions`：排除 `_test.go` 和 `vendor`SonarQube
- `sonar.qualitygate.wait=true`：**阻塞到质量门禁出结果**SonarQube
- `allow_failure: false`：代码不达标则 CI 失败（推荐）

------

### 四、GitLab MR 集成（最佳实践）

#### 1. 开启 MR 装饰（Sonar 评论）

SonarQube → **Administration → DevOps Platform Integrations → GitLab**

- 填入 GitLab URL、管理员 Token
- 项目 → **Project Settings → DevOps Platform Integration** → 绑定 GitLab 项目SonarQube

效果：

- MR 自动显示：✅ Passed / ❌ Failed
- 直接在代码行标注 Bug、漏洞、坏味道

#### 2. 阻止不合格代码合并

GitLab → **Settings → Merge requests → Merge checks**

- 勾选 **Pipelines must succeed**
- 勾选 **All threads must be resolved**SonarQube

------

### 五、常见问题与排错

1. **Sonar 显示 0 文件 / 0 覆盖率**

   

   - 确认 `sonar.sources=.`、`go.mod` 在根目录
   - 检查 `coverage.out` 存在且非空
   - 路径正确：`sonar.go.coverage.reportPaths=coverage.out`SonarQube

   

2. **权限错误 / 连接失败**

   

   - 确认 `SONAR_TOKEN` 正确、已 Mask
   - 确认 GitLab Runner 能访问 `SONAR_HOST_URL`

   

3. **覆盖率偏低**

   

   - 运行 `go tool cover -html=coverage.out` 本地查看
   - 确认测试覆盖核心业务逻辑

   

4. **Sonar 扫描慢**

   

   - 开启缓存：`.sonar/cache/`
   - 排除 `vendor/`、`dist/`、`testdata/`

   

------

### 六、完整效果

- GitLab CI 流水线：**Test → Sonar**
- 测试报告：GitLab → **Build → Test report**
- 质量报告：Sonar 面板 + **GitLab MR 评论**
- 门禁：质量不达标 → **合并被阻止**

```
# 定义流水线阶段

  - scan

# 缓存加速

    - .trivy/

# ======================
# 1. 构建 Go 项目镜像
# ======================
build-go-image:
  stage: build
  image: docker:24-dind
  services:
    - docker:24-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $FULL_IMAGE .
    - docker push $FULL_IMAGE
  only:
    - main
    - merge_requests

# ======================
# 2. TRIVY 镜像安全扫描（核心）
# ======================
trivy-image-scan:
  stage: scan
  image: aquasec/trivy:latest
  dependencies: []
  script:
    # 登录镜像仓库（必须）
    - trivy login $CI_REGISTRY -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD
    
    # 扫描镜像 → 生成 GitLab 兼容的安全报告
    - trivy image
        --format gitlab
        --output gl-container-scanning-report.json
        --severity HIGH,CRITICAL       # 只拦截高危/严重漏洞
        --exit-code 1                   # 发现漏洞 → 流水线失败（门禁）
        --no-progress
        $FULL_IMAGE

  # 始终上传报告（即使扫描失败）
  artifacts:
    reports:
      container_scanning: gl-container-scanning-report.json
    paths:
      - gl-container-scanning-report.json
    when: always

  only:
    - main
    - merge_requests
```

## 三、关键配置解释（你必须懂）

### 1. 门禁规则

yaml











```
--severity HIGH,CRITICAL   # 只拦截高危/严重漏洞
--exit-code 1              # 发现就失败，阻止合并
```

- 想宽松一点：去掉 `--exit-code 1`
- 想全级别扫描：`--severity UNKNOWN,LOW,MEDIUM,HIGH,CRITICAL`

### 2. GitLab 集成报告

扫描后会自动在 GitLab 显示：

- 项目 → **安全与合规 → 容器扫描**
- 合并请求 → 直接显示漏洞数量

### 3. 支持扫描 Go 代码依赖

Trivy 会自动扫描 `go.mod` / `go.sum` 里的第三方库漏洞，**不需要额外配置**。

### 一、两种集成方式（任选其一）

#### 方式 1：使用 GitLab 内置容器扫描模板（最简）

GitLab 自带 Trivy 扫描模板，**一行代码启用**极狐GitLab：

yaml











```
# .gitlab-ci.yml
include:
  - template: Security/Container-Scanning.gitlab-ci.yml  # 官方模板

variables:
  # 要扫描的镜像（GitLab 容器仓库）
  CS_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
  # 扫描级别：只拦截高危/严重
  CS_SEVERITY_THRESHOLD: HIGH,CRITICAL
  # 使用 Trivy 作为扫描器
  CS_ANALYZER_IMAGE: aquasec/trivy:latest

# 可选：覆盖默认行为
container_scanning:
  stage: security
  needs: [build]  # 依赖构建任务
  allow_failure: false  # 发现漏洞则流水线失败
```

#### 方式 2：完全自定义 Trivy 任务（推荐，可控性强）

适合需要精细控制、私有镜像、缓存优化的场景：

yaml











```
stages:
  - build
  - security

# 全局变量
variables:
  IMAGE_NAME: my-go-app
  IMAGE_TAG: $CI_COMMIT_SHORT_SHA
  FULL_IMAGE: $CI_REGISTRY/$CI_PROJECT_PATH/$IMAGE_NAME:$IMAGE_TAG
  TRIVY_CACHE_DIR: .trivy-cache
  TRIVY_NO_PROGRESS: "true"

# 1. 构建镜像（Go 项目示例）
build:
  stage: build
  image: docker:24-dind
  services:
    - docker:24-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $FULL_IMAGE .
    - docker push $FULL_IMAGE
  only: [main, merge_requests]

# 2. Trivy 镜像扫描（核心）
trivy-scan:
  stage: security
  image: aquasec/trivy:latest
  needs: [build]
  variables:
    TRIVY_USERNAME: $CI_REGISTRY_USER
    TRIVY_PASSWORD: $CI_REGISTRY_PASSWORD
    TRIVY_AUTH_URL: $CI_REGISTRY
  script:
    # 预热漏洞库
    - trivy image --download-db-only
    # 生成 GitLab 兼容报告
    - trivy image
        --format gitlab
        --output gl-container-scanning-report.json
        --severity HIGH,CRITICAL
        --exit-code 1  # 漏洞 → 失败
        --ignore-unfixed  # 忽略未修复漏洞（可选）
        $FULL_IMAGE
  # 缓存加速
  cache:
    paths:
      - $TRIVY_CACHE_DIR
  # 报告上传 GitLab
  artifacts:
    reports:
      container_scanning: gl-container-scanning-report.json
    when: always
  only: [main, merge_requests]
```

------

### 二、关键参数说明（必懂）

- `--severity HIGH,CRITICAL`：只拦截高危 / 严重漏洞
- `--exit-code 1`：发现漏洞 → 流水线失败（质量门禁）
- `--ignore-unfixed`：忽略暂无修复方案的漏洞（避免误阻塞）
- `TRIVY_CACHE_DIR`：缓存漏洞库，**大幅提速**
- `artifacts.reports.container_scanning`：报告自动同步到 GitLab 安全面板
