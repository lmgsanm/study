要让 kube-prometheus 监控 ArgoCD 指标，核心是创建 **ServiceMonitor** 资源，并确保其标签与 Prometheus 匹配、选择器能准确定位 ArgoCD 的 metrics Service。

### 一、确认前提

1. **ArgoCD metrics Service 已存在**
   bash

   运行

   ```
   kubectl get svc argocd-server-metrics -n argocd
   ```

   * 名称：`argocd-server-metrics`
   * 命名空间：`argocd`
   * 端口名：`metrics`（通常对应 8083 端口）
2. **Prometheus 识别 ServiceMonitor 的标签**

   查看 kube-prometheus 中 Prometheus 资源的 `serviceMonitorSelector`：
   bash

   运行

   ```
   kubectl get prometheus k8s -n monitoring -o jsonpath='{.spec.serviceMonitorSelector}'
   ```

   通常为 `release: kube-prometheus` 或 `release: prometheus-operator`。

### 二、创建 ServiceMonitor

在 `monitoring` 命名空间下创建 `argocd-servicemonitor.yaml`：

yaml

```
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: argocd-server-metrics
  namespace: monitoring  # 与 Prometheus 同Namespace
  labels:
    # 关键：必须与 Prometheus.serviceMonitorSelector 匹配
    release: kube-prometheus
spec:
  # 从 argocd 命名空间发现服务
  namespaceSelector:
    matchNames:
      - argocd
  # 匹配 argocd-server-metrics Service 的标签
  selector:
    matchLabels:
      app.kubernetes.io/name: argocd-server-metrics
  # 配置抓取端点
  endpoints:
    - port: metrics       # 必须与 Service 端口名一致
      path: /metrics     # 指标路径（默认）
      interval: 30s      # 抓取间隔
      scrapeTimeout: 10s # 超时时间
```

### 三、应用配置

bash

运行

```
kubectl apply -f argocd-servicemonitor.yaml
```

### 四、验证

1. **检查 ServiceMonitor 状态**
   bash

   运行

   ```
   kubectl get servicemonitor argocd-server-metrics -n monitoring
   ```
2. **在 Prometheus UI 确认目标**

   * 访问 Prometheus UI → **Status → Targets**
   * 应能看到 `monitoring/argocd-server-metrics/0` 状态为 **UP**

### 五、（可选）监控所有 ArgoCD 组件

ArgoCD 还有其他 metrics Service（`argocd-metrics`、`argocd-repo-server-metrics` 等），可创建完整的 ServiceMonitor：

yaml

```
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: argocd-metrics
  namespace: monitoring
  labels:
    release: kube-prometheus
spec:
  namespaceSelector:
    matchNames:
      - argocd
  selector:
    matchLabels:
      app.kubernetes.io/part-of: argocd
  endpoints:
    - port: metrics
      path: /metrics
      interval: 30s
```

## 三、步骤 2：创建 Secret 存放额外抓取配置

kube-prometheus 支持通过 `additionalScrapeConfigs` 加载外部抓取规则。

新建文件 `prometheus-additional-scrape.yaml`：

yaml

```
apiVersion: v1

metadata:
  name: prometheus-additional-scrape
  namespace: monitoring
type: Opaque
stringData:
  additional-scrape-configs.yaml: |
    - job_name: "argocd-external"
      static_configs:
        - targets:
          - "192.168.1.100:8083"   # 替换成你外部 ArgoCD 的 metrics 地址
      metrics_path: /metrics       # 默认路径
      scrape_interval: 30s
      scrape_timeout: 10s
```

应用：

bash

运行

```
kubectl apply -f prometheus-additional-scrape.yaml
```

# 二、修改 Filebeat 配置（关键步骤）

Filebeat 用 **ConfigMap** 管理配置，我们只需要加一段 `filebeat.inputs` 或修改 `kubernetes` 采集器，只收集 `argocd` 命名空间的日志。

## 1. 编辑 Filebeat ConfigMap

bash

运行

```
kubectl edit configmap filebeat -n elk
```

找到 `filebeat.yml` 里的 `filebeat.inputs` 或 `filebeat.autodiscover` 部分。

## 2. 添加采集 argocd 命名空间日志的配置

### 方案 A：使用 autodiscover（推荐，最通用）

在 `filebeat.autodiscover` 下添加：

yaml

```
filebeat.autodiscover:
  providers:
    - type: kubernetes
      hints.enabled: true
      hints.default_config.enabled: false
      include_namespaces:
        - argocd
      templates:
        - condition:
            equals:
              kubernetes.namespace: argocd
          config:
            - type: container
              paths:
                - /var/log/containers/*${data.kubernetes.container.id}.log
              multiline.type: pattern
              multiline.pattern: '^[[:space:]]+(at|\.{3})|^Caused|^org|^java|^time="'
              multiline.negate: false
              multiline.match: after
```

如果你不想这么复杂， **最简单可用版本** ：

yaml

```
filebeat.inputs:
  - type: container
    paths:
      - /var/log/containers/*_argocd_*.log
    processors:
      - add_kubernetes_metadata:
          host: ${NODE_NAME}
          matchers:
            - logs_path:
                logs_path: "/var/log/containers/"
```

> 说明：
>
> `/var/log/containers/*_argocd_*.log` 是 K8s 软链接路径，专门匹配 argocd 命名空间下所有容器日志。

## 3. 确保输出到 Elasticsearch

你的配置里本来就有，确认类似这段即可：

yaml

```
output.elasticsearch:
  hosts: ["elasticsearch.elk.svc.cluster.local:9200"]
  # username: ...
  # password: ...
```

ConfigMap 自动保存。

# 五、进阶：多行日志堆栈合并（可选但强烈建议）

ArgoCD 很多 Java/Go 错误日志会换行，加上 multiline 配置：

```

multiline.match: after
```

这样完整错误堆栈会被合并成一条日志，方便排查。

# 三、配置 GitLab 对接 Casdoor（OAuth2 登录）

你的 GitLab 使用 Helm 部署在 `gitl` 命名空间。

## 1. 导出当前 GitLab Values

bash

运行

```bash
helm get values gitlab -n gitl -o yaml > gitlab-values.yaml
```

## 2. 在 Values 中添加 Casdoor OAuth 配置

在 `global.appConfig` 下添加：

yaml

```yaml
global:
  appConfig:
    omniauth:
      enabled: true
      allowSingleSignOn: ["casdoor"]
      autoLinkUser: true
      blockAutoCreatedUsers: false
      providers:
        - {
            name: "casdoor",
            label: "Casdoor 统一登录",
            app_id: "【替换casdoor-gitlab-clientID】",
            app_secret: "【替换casdoor-gitlab-clientSecret】",
            strategy_class: "OmniAuth::Strategies::OpenIDConnect",
            args: {
              scope: "openid profile email",
              response_type: "code",
              issuer: "https://casdoor.example.com",
              discovery: true
            }
          }
```

## 3. 更新 GitLab

bash

运行

```bash
helm upgrade gitlab gitlab/gitlab -f gitlab-values.yaml -n gitl
```

## 4. 重启 GitLab Web 服务

bash

运行

```bash
kubectl rollout status deployment gitlab-webservice -n gitl
```

打开 `https://gitlab.example.com`

登录按钮下方出现：**Casdoor 统一登录**
